The taxonomy drives two revenue core outcomes through Sailthru

Both depend on tagged articles in Sailthru's Content Library. Both must survive the Arc migration.
Walk through the flow from top to bottom. An article gets tagged in the CMS. Those tags flow to Sailthru as "interests" via the page scraper. From there, two things happen. First, Topic Takeovers: Sailthru builds interest scores for each user based on which tagged articles they read over time. Sponsor audience lists filter users who score above a threshold on clusters of tags. This is the revenue mechanism - direct-sold sponsor campaigns. Second, My Topics: users explicitly follow topics, and each day the system matches published articles against their follows and sends a personalized email. The engagement is exceptional -3x the flagship newsletter. Both depend on the same thing: correctly tagged articles landing in Sailthru's Content Library. That's why the taxonomy migration matters.

In Arc, two primitives replace one - but Sailthru works the same way

Today (Drupal)

One flat taxonomy of ~3,000 terms handles everything - editorial navigation, ad targeting, user follows, newsletter content matching.

All terms pass to Sailthru as interests. No distinction between a section like "sequencing" and a targeting tag like "liquid biopsy."

The vocabulary is ungoverned. Any editor can create tags. No lifecycle management.

Tomorrow (Arc)

Sections (~210 nodes) handle editorial hierarchy - navigation, category pages, breadcrumbs. Per-site, not shared across brands.

Tags (~275-325 governed terms, potentially more pending ad ops input and company name scoping) handle cross-cutting targeting - sponsor campaigns, niche topics that span sections, entity names.

Both sections and tags pass to Sailthru as interests. Downstream, nothing changes - interest scoring, audience building, and content matching all work the same way.

The key message: Sailthru doesn't care whether a term is a "section" or a "tag" in Arc. It just needs strings in the Content Library to score against. So both sections and tags will pass to Sailthru as interests, exactly like the flat taxonomy does today. What changes is the CMS side - instead of one unstructured list of 3,000 terms, editors work with a structured section hierarchy for editorial navigation and a governed set of roughly 300 tags for targeting. The Sailthru plumbing stays intact. The segment rules will need rebuilding to match new term names, but the mechanics are identical. Migration planning has already mapped most of the existing taxonomy to this new structure.

Some terms can't be sections - they power ad targeting across editorial boundaries

  • Cross-cutting topics - terms like "liquid biopsy" or "companion diagnostics" span multiple editorial sections. Sections are hierarchical; tags let the same article build interest scores across multiple targeting contexts.
  • Entity names - sponsor companies aren't editorial categories. They're targeting handles that enable 100% share-of-voice campaigns. Users who read Illumina-tagged articles build Illumina interest scores, which feed the Illumina audience list.
  • Ad-ops targeting clusters - Topic Takeover campaigns target groups of 13-21 tags per audience list, cutting across the section tree by design. ~130 tags are in active segment rules across 16 audience lists. Sends perform at 10.5% open / 2.6% CTR.
  • The governed vocabulary is smaller than assumed - data across three systems sizes the load-bearing taxonomy at ~275-325 terms, not the ~3,000 in Drupal. ~180 terms show zero evidence of use and are retirement candidates.

Outstanding: Ad ops input on active targeting tags and company name inclusion could both increase the governed vocabulary. Final number depends on cross-referencing both against what's already counted.

Tags and sections solve different problems. The architecture needs both.
This is the structural argument for why tags exist alongside sections. Sections organize content into an editorial hierarchy - an article lives in one section. Tags enable targeting across those boundaries. A single article about Illumina's new sequencing platform could build interest scores for the sequencing audience, the Illumina sponsor list, and a liquid biopsy campaign simultaneously. That cross-cutting capability is what sponsors are buying. The data sizing is important: roughly 275-325 terms are actually load-bearing, not the 3,000 historically accumulated. About 180 terms have zero evidence of use. Two outstanding items: ad ops hasn't audited which tags are in active GAM targeting yet, and the company names question is still open. Both need resolution before we lock the vocabulary. The takeaway sets up the options: tag assignment should be automated with AI classification.

Both paths lead to AI-driven classification - the question is sequencing

Option A: Governed Tags, AI Later
  • Load ~275-325+ tags + ~210 sections into Arc
  • Both pass to Sailthru as interests - downstream unchanged
  • Editors tag manually at launch
  • AI classification pipeline layers on post-launch
  • Lower cost, lower risk, proven pattern
Option B: AI Classification in Parallel
  • Same June 18 migration work as Option A
  • Build AI classification pipeline in parallel starting now
  • Pipeline assigns tags from the governed vocabulary
  • Editors confirm or override AI suggestions
  • Faster to end-state, higher upfront investment
If the parallel track slips, Option B degrades gracefully to Option A.
Both options end up in the same place: AI classifies articles against GNW's vocabulary, manual tagging becomes override-only. The Sailthru pipeline is identical in both - sections and tags pass as interests, interest scoring works the same way, audience lists filter the same way. Option A gets to AI incrementally. Option B front-loads the investment. The critical safety net: if Option B's parallel track slips, you're in Option A anyway. The June 18 work is identical. The downstream Sailthru mechanics are identical. The only question is when AI classification arrives and how much engineering capacity to invest now versus later.

The option choice affects what runs in parallel and when AI arrives - not the cutover itself.
Ground the room: the bulk of the work is the same regardless of which option we go with. Tags get seeded, sections get built, segment rules get rebuilt to match new names, content gets migrated. The critical validation is end-to-end: a tagged article in Arc needs to reach the Sailthru Content Library, generate correct interest scores, and populate the right audience lists. That's the same pipeline in both options. The only difference is whether we also invest in building the AI classification pipeline in parallel or start that post-launch.

Outstanding questions and decisions

  • Vocabulary lock - the ~275-325 estimate needs ad ops input on active targeting tags and a decision on which company names to include. Both could move the number up. Must finalize before tag seeding begins.
  • Sailthru rebuild scope - ~16 audience list segment rules need rewriting to match new term names. Templates and send logic referencing the current taxonomy format need auditing. The My Topics Daily content matching logic needs to be confirmed so it keeps working with the new naming.
  • Ad ops cutover validation - GAM line items targeting current term names need auditing and updating where names change. Sponsor campaigns can't regress at cutover.
  • AI classification sequencing - the June 18 migration work is identical in both options. The choice is whether to invest in building the classification pipeline in parallel or layer it on after launch.
